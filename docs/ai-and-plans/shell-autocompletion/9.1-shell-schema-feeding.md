## Summary

Shell query results now feed documents to SchemaStore, enabling field name completions in the shell after running queries like `db.users.find()`. Previously, schema data was only populated from the Collection View and Query Playground â€” the shell discarded all query result documents without analyzing them.

## Problem

When a user runs `db.users.find()` in the interactive shell and then types `db.users.find({ <Tab>`, no field names appear. The SchemaStore has no data for that collection because the shell evaluation path never fed documents to it.

The Query Playground solved this same problem in Step 6.1 by calling `feedResultToSchemaStore()` after each evaluation. The shell was missing this integration because:

1. The worker serializes `result.printable` to an EJSON **string** for IPC transfer
2. `DocumentDBShellPty` received the string and passed it directly to the formatter for display
3. The raw documents were never deserialized back to objects for schema analysis

### Data flow comparison

| Stage | Playground | Shell (before) | Shell (after) |
|-------|-----------|----------------|---------------|
| Worker â†’ main thread | EJSON string | EJSON string | EJSON string |
| Deserialization | `PlaygroundEvaluator.deserializeResult()` | â€” (skipped) | `deserializeResultForSchema()` |
| Schema feeding | `feedResultToSchemaStore()` | â€” (skipped) | `feedResultToSchemaStore()` |
| Display formatting | `formatResult()` | `ShellOutputFormatter.formatResult()` | `ShellOutputFormatter.formatResult()` |

## Solution

### 1. Shared utility extraction

Extracted `feedResultToSchemaStore()`, `deserializeResultForSchema()`, and `randomSample()` from `executePlaygroundCode.ts` into a shared module at `src/documentdb/feedResultToSchemaStore.ts`. Both the playground and the shell now import from this shared utility.

**Key behaviors preserved from the playground implementation:**

| Behavior | Why |
|----------|-----|
| Only feed `Cursor` and `Document` result types | Other types (string, null, exit, clear) don't contain documents |
| Require `source.namespace.collection` | Can't store schema without knowing the collection |
| Unwrap CursorIterationResult `{ cursorHasMore, documents }` | @mongosh wraps cursor batches in this structure |
| Filter to objects with `_id` | Rejects primitives, arrays, and `_id: 0` projections |
| Cap at 100 documents (random sample) | Prevents excessive schema analysis on large result sets |
| Canonical EJSON parsing (`relaxed: false`) | Preserves BSON types (ObjectId, Date, Decimal128) for accurate type detection |

### 2. Shell PTY integration

Added `maybeFeedSchemaStore()` to `DocumentDBShellPty`, called after `maybeWriteActionLine()` in `evaluateInput()`. The method:

1. Checks result type is `Cursor` or `Document` with a namespace (fast guard)
2. Asynchronously deserializes the EJSON string via `deserializeResultForSchema()`
3. Feeds the deserialized result to `feedResultToSchemaStore()`
4. Catches and silently ignores any errors (non-critical, best-effort)

Schema feeding is **non-blocking** â€” the method fires and forgets. The prompt appears immediately after output; deserialization and schema analysis happen in the background.

### 3. Also in this PR: Completion improvements

Two completion quality improvements were also included:

- **Field/operator separation:** At `key` position inside `find({})`, only field names are shown. Operators appear only when `$` is typed. This eliminates the "random operator dump" on empty prefix.
- **Nested field prefix fix:** `extractArgumentPrefix` and `findReplacementStart` now include `.` in their scan pattern, so `toplevel.is<cursor>` correctly matches `toplevel.isEnabled` instead of `ISODate`.

## Testing

- **26 unit tests** for the shared `feedResultToSchemaStore` utility covering:
  - Result type filtering (Cursor, Document, string, null, exit, clear)
  - Namespace validation (missing namespace, empty collection)
  - Printable extraction (arrays, single docs, CursorIterationResult unwrapping)
  - Document filtering (primitives, nested arrays, missing `_id`)
  - Document cap (100 limit with random sampling)
  - EJSON deserialization with fallback chain

- Playground regression: verified existing playground tests still pass after the extraction refactor.
