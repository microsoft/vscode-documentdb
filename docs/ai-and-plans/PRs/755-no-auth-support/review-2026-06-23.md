## Findings

### High: Updating a connection to NoAuth can leave stale native or Entra secrets in storage

**Where:** [src/commands/updateCredentials/ExecuteStep.ts](../../../../src/commands/updateCredentials/ExecuteStep.ts#L57-L81), [src/documentdb/CredentialCache.ts](../../../../src/documentdb/CredentialCache.ts#L283-L301)

The update-credentials save step handles Native and Microsoft Entra ID, but it has no branch for `AuthMethodId.NoAuth`. If a connection has existing `nativeAuthConfig` or `entraIdAuthConfig`, then the user changes the selected method to `NoAuth`, the code still updates `properties.selectedAuthMethod` but leaves those older auth-specific secrets untouched.

That stale state is not just cosmetic. `CredentialCache.setFromConnectionItem` correctly preserves the explicit `NoAuth` auth mechanism, but it still copies `secrets.entraIdAuthConfig` and reads `secrets.nativeAuthConfig` into local variables before calling `setAuthCredentials`. Other code that asks the cache for user/password, such as migration sharing, can therefore observe credentials on a connection now marked as NoAuth. At minimum this makes the stored item internally contradictory; in the worst case it reintroduces credential leakage into paths this PR is trying to make credential-free.

**Recommendation:** Add an explicit NoAuth branch in `updateCredentials` that clears both `nativeAuthConfig` and `entraIdAuthConfig`, and add tests for switching Native -> NoAuth and Entra -> NoAuth.

> **Independent assessment — verdict: CONFIRMED. Severity: High (functional, not just cosmetic).**
>
> I traced the full path and the finding is real:
>
> 1. [updateCredentials/ExecuteStep.ts](../../../../src/commands/updateCredentials/ExecuteStep.ts#L57-L81) has branches only for `NativeAuth`, `MicrosoftEntraID` (with config) and `MicrosoftEntraID` (without config). There is **no** `NoAuth` branch, so when the user switches to NoAuth the embedded username/password are stripped from the connection string (lines 44-45) **but** `nativeAuthConfig` / `entraIdAuthConfig` in `secrets` are left intact.
> 2. On the next load, [CredentialCache.setFromConnectionItem](../../../../src/documentdb/CredentialCache.ts#L283-L301) respects the explicit `NoAuth` mechanism, yet still reads `secrets.nativeAuthConfig` into `username`/`password` and copies `secrets.entraIdAuthConfig`, then calls `setAuthCredentials`. That re-injects the stale username/password into `connectionStringWithPassword` and repopulates `nativeAuthConfig`/`entraIdConfig` in the cache.
> 3. Migration sharing in [accessDataMigrationServices.ts](../../../../src/commands/accessDataMigrationServices/accessDataMigrationServices.ts#L116-L123) then reads `CredentialCache.getConnectionUser/getConnectionPassword` (both back the value out of `nativeAuthConfig`) and rebuilds a credential-bearing connection string. So a connection the user explicitly marked **NoAuth** can still connect/share with the old secrets. This is a correctness + credential-persistence bug, so I agree with High — though note it requires the specific Native/Entra → NoAuth transition and is not a remote exploit.
>
> **Suggested solution (recommended): clear configs at the write boundary.** Add a `NoAuth` branch in `updateCredentials/ExecuteStep` that sets both `nativeAuthConfig` and `entraIdAuthConfig` to `undefined`. Better: invert the logic so the save explicitly writes only the config for the selected method and clears all others (a small `switch (authMethod)` that always assigns all three fields). Complexity: **Low** (≈10 lines in one file + 2 regression tests).
> - _Pros:_ fixes the root cause where data is persisted; storage becomes internally consistent; no downstream reader needs to change; easy to test deterministically.
> - _Cons:_ does not retroactively clean connections already saved as NoAuth-with-stale-secrets before the fix (acceptable; could add a one-time sanitize on load if needed).
>
> **Alternative (defense in depth): also gate the cache read.** In `setFromConnectionItem`, when `selectedAuthMethod === NoAuth`, skip reading `nativeAuthConfig`/`entraIdAuthConfig` entirely. Complexity: **Low**.
> - _Pros:_ protects against any other writer that leaves stale secrets; covers already-corrupted storage.
> - _Cons:_ treats the symptom not the cause; leaves contradictory data on disk; ideally done *in addition to*, not instead of, the write-side fix.
>
> I'd do both: the write-side clear is mandatory, the cache-side guard is a cheap safety net.

**Resolution (commit `e936fe3c`):** Did both, as planned.
- Write-side: added an explicit `AuthMethodId.NoAuth` branch in [updateCredentials/ExecuteStep.ts](../../../../src/commands/updateCredentials/ExecuteStep.ts) that sets `nativeAuthConfig` and `entraIdAuthConfig` to `undefined`.
- Cache-side defense in depth: [CredentialCache.setFromConnectionItem](../../../../src/documentdb/CredentialCache.ts) now computes `isNoAuth` and, for an explicit NoAuth method, skips reading `nativeAuthConfig`/`entraIdAuthConfig` so stale secrets in already-saved records can never be re-surfaced.
- Tests: new [updateCredentials/ExecuteStep.test.ts](../../../../src/commands/updateCredentials/ExecuteStep.test.ts) covers Native→NoAuth and Entra→NoAuth clearing; [CredentialCache.test.ts](../../../../src/documentdb/CredentialCache.test.ts) gains a case asserting stale native/Entra secrets are never surfaced for a NoAuth item.

### Medium: New-connection save can persist stale Entra config for Native or NoAuth

**Where:** [src/commands/newConnection/ExecuteStep.ts](../../../../src/commands/newConnection/ExecuteStep.ts#L195-L201)

The new-connection execute step now correctly persists native credentials only when `selectedAuthenticationMethod === NativeAuth`, but `entraIdAuthConfig` is still stored unconditionally from the wizard context. If the wizard context contains stale Entra data after the user changes auth method, a Native or NoAuth connection can be saved with Entra-specific config attached.

This matches Copilot review thread `PRRT_kwDOODtcO86Ln9xO`. The same principle applied to native credentials should apply here too: persist Entra config only when the selected method is `MicrosoftEntraID`; otherwise clear it.

**Recommendation:** Change the storage item to set `entraIdAuthConfig` only for `AuthMethodId.MicrosoftEntraID`, and add a regression test for Entra -> NoAuth or Entra -> Native after backtracking/changing auth method.

> **Independent assessment — verdict: CONFIRMED. Severity: Medium (downgrade to Low if the wizard always clears `entraIdAuthConfig` on method change).**
>
> In [newConnection/ExecuteStep.ts](../../../../src/commands/newConnection/ExecuteStep.ts#L195-L201) the storage item gates native creds (`nativeAuthConfig: usesNativeCredentials ? context.nativeAuthConfig : undefined`) but writes `entraIdAuthConfig: context.entraIdAuthConfig` unconditionally. So the asymmetry the reviewer describes is real. Whether stale Entra data is actually present depends on whether the wizard resets `context.entraIdAuthConfig` when the user backtracks and changes method — I did not find code that clears it, so the risk is plausible. Lower blast radius than the High finding because Entra config is tenant/subscription metadata, not a password, but it still produces a contradictory NoAuth/Native record.
>
> **Suggested solution: mirror the native-cred gate.** Compute `usesEntraId = selectedAuthenticationMethod === AuthMethodId.MicrosoftEntraID` and write `entraIdAuthConfig: usesEntraId ? context.entraIdAuthConfig : undefined`. Complexity: **Low** (1 line + 1 test). Ideally factor both gates into a single "build secrets for selected method" helper so new auth methods can't reintroduce the asymmetry.
> - _Pros:_ symmetric with the native fix; trivial; closes the gap the reviewer flagged in thread `PRRT_kwDOODtcO86Ln9xO`.
> - _Cons:_ if some flow legitimately relies on carrying Entra metadata across a NoAuth save (none found), this would drop it — unlikely, but worth a quick grep before merging.

**Resolution (commit `44d75f09`):** Added `usesEntraId = newAuthenticationMethod === AuthMethodId.MicrosoftEntraID` in [newConnection/ExecuteStep.ts](../../../../src/commands/newConnection/ExecuteStep.ts) and changed the stored secret to `entraIdAuthConfig: usesEntraId ? context.entraIdAuthConfig : undefined`, mirroring the existing native-cred gate. Added two regression tests in [newConnection/ExecuteStep.test.ts](../../../../src/commands/newConnection/ExecuteStep.test.ts): stale Entra config is dropped when the method is changed to NoAuth and to Native. Replied in Copilot thread `PRRT_kwDOODtcO86Ln9xO`.

### Medium: New tree-expansion telemetry uses a differently-cased connectionResult

**Where:** [src/tree/documentdb/ClusterItemBase.ts](../../../../src/tree/documentdb/ClusterItemBase.ts#L232-L238), [src/tree/documentdb/ClusterItemBase.test.ts](../../../../src/tree/documentdb/ClusterItemBase.test.ts#L164-L171)

The new `listDatabases()` failure path records `connectionResult = 'Failed'`. Existing connection telemetry appears to use lowercase outcome values such as `success`, `failed`, and `cancelled`, so this will fragment dashboards and queries that group by `connectionResult`.

This matches Copilot review thread `PRRT_kwDOODtcO86Ln9wH`. The test currently locks in the uppercase value, so the test should change with the implementation.

**Recommendation:** Use `connectionResult = 'failed'` and update the new test expectation.

> **Independent assessment — verdict: CONFIRMED. Severity: Low (telemetry hygiene; downgraded from Medium).**
>
> I grepped every `connectionResult` assignment in the codebase. All other call sites use lowercase — `'success'`, `'failed'`, `'cancelled'` (e.g. [MongoRUResourceItem.ts](../../../../src/plugins/service-azure-mongo-ru/discovery-tree/documentdb/MongoRUResourceItem.ts#L133), [DocumentDBResourceItem.ts](../../../../src/plugins/service-azure-mongo-vcore/discovery-tree/documentdb/DocumentDBResourceItem.ts#L182), and the sibling `'cancelled'` on [ClusterItemBase.ts](../../../../src/tree/documentdb/ClusterItemBase.ts#L181) just 50 lines above). Only the new line [ClusterItemBase.ts](../../../../src/tree/documentdb/ClusterItemBase.ts#L234) uses `'Failed'`. So the inconsistency is real and will split any group-by on this dimension. I rate it **Low** rather than Medium: it's analytics-only, no user or runtime impact, and trivially fixed.
>
> **Suggested solution:** change `'Failed'` → `'failed'` and update the locked-in expectation in [ClusterItemBase.test.ts](../../../../src/tree/documentdb/ClusterItemBase.test.ts#L164-L171). Complexity: **Trivial** (2 lines). Note this is a normal (non-`TDD:`) test, so updating the expectation alongside the implementation is appropriate.
> - _Pros:_ one-character semantic fix; keeps dashboards clean; no API surface change.
> - _Cons:_ none of note. (If historical dashboards already special-case the uppercase value, they'd need updating — but standardizing now prevents future drift.)

**Resolution (commit `87a258c7`):** Changed `'Failed'` → `'failed'` in [ClusterItemBase.ts](../../../../src/tree/documentdb/ClusterItemBase.ts) and updated the locked-in expectation in [ClusterItemBase.test.ts](../../../../src/tree/documentdb/ClusterItemBase.test.ts). Replied in Copilot thread `PRRT_kwDOODtcO86Ln9wH`.

### Low: Shell auth label localization is inconsistent

**Where:** [src/documentdb/shell/DocumentDBShellPty.ts](../../../../src/documentdb/shell/DocumentDBShellPty.ts#L486-L493)

The new NoAuth shell banner label is localized, while the adjacent `Entra ID` and `SCRAM` labels remain raw string literals. These labels are rendered in the integrated shell output through the localized `Authentication: {1}` message, so they are user-facing.

This matches Copilot review thread `PRRT_kwDOODtcO86Ln9wy`. The pre-existing literals are not introduced by this PR, but the touched expression now mixes localized and non-localized labels.

**Recommendation:** Wrap the `Entra ID` and `SCRAM` labels in `l10n.t(...)` as well, or use localized labels from the auth-method metadata where appropriate.

> **Independent assessment — verdict: CONFIRMED. Severity: Low (cosmetic/i18n consistency).**
>
> Confirmed at [DocumentDBShellPty.ts](../../../../src/documentdb/shell/DocumentDBShellPty.ts#L486-L493): the ternary returns a localized `l10n.t('No Authentication')` for NoAuth but bare `'Entra ID'` and `'SCRAM'` literals for the other branches, and the result is interpolated into a localized `Connected to`/auth summary line, so all three are user-facing. The mixed localization is real. `SCRAM` is arguably a protocol acronym that shouldn't be translated, but `Entra ID` is a product label that other locales may transliterate.
>
> **Suggested solution (preferred): source labels from auth-method metadata.** If the `AuthMethod` registry already exposes a localized display name per method, use it here instead of an inline ternary — that removes the literal-vs-localized split structurally and keeps NoAuth/Native/Entra labels consistent everywhere. Complexity: **Low–Medium** depending on whether such metadata exists.
> - _Pros:_ single source of truth for auth labels; future auth methods get labels for free; no per-call-site l10n drift.
> - _Cons:_ slightly more plumbing; need to confirm the metadata is available in this context.
>
> **Fallback: wrap inline.** Just wrap `'Entra ID'` (and optionally `'SCRAM'`) in `l10n.t(...)`. Complexity: **Trivial**.
> - _Pros:_ one-line-per-label; matches the reviewer's literal recommendation.
> - _Cons:_ leaves three independent label sources that can drift again; doesn't fix the same pattern if it exists elsewhere.
>
> Either is fine for this PR; I'd not block merge on it.

**Resolution (commit `91468c3d`):** Per request, used the simple fallback — wrapped both `'Entra ID'` and `'SCRAM'` in `l10n.t(...)` in [DocumentDBShellPty.ts](../../../../src/documentdb/shell/DocumentDBShellPty.ts) so all three auth labels in the shell banner are localized consistently. Ran `npm run l10n` to update the bundle. Replied in Copilot thread `PRRT_kwDOODtcO86Ln9wy`.
