### F1 ŌĆö Stage 3 completion telemetry marks successful streams as aborted

**Original severity:** High
**My severity:** **High (confirmed)** ŌĆö this corrupts the canonical Stage 3 telemetry event that the PR explicitly designates as the replacement for the auto rpc event. Every successful stream is currently logged with `aborted='true'`, which means dashboards cannot distinguish success from user-cancel or shutdown-cancel. The fact that the surrounding code documents this event as the source of truth makes the bug worse, not less severe.

**Where:** [src/webviews/documentdb/collectionView/queryInsights/queryInsightsEventsRouter.ts](../../../../src/webviews/documentdb/collectionView/queryInsights/queryInsightsEventsRouter.ts#L155-L185), [src/webviews/documentdb/collectionView/queryInsights/queryInsightsEventsRouter.ts](../../../../src/webviews/documentdb/collectionView/queryInsights/queryInsightsEventsRouter.ts#L555-L572)

**Verification:** Confirmed against the current source. The `finally` block runs `abortController.abort()` unconditionally, then calls `flushCompletionEvent()`, which samples `abortController.signal.aborted`. Because both run synchronously inside the same `finally`, the flag is always observed as `true` regardless of how the generator exited (normal `return`, framework-driven `iterator.return()`, host shutdown, or an actually-cancelled request).

**Proposed solutions:**

1. **Capture the real abort cause before cleanup** (recommended).
   Track an explicit `cancelCause` (`'none' | 'ctx' | 'cleanup'`) set in the `onCtxAbort` listener and in any `return`-on-abort branch. The `finally` then computes `aborted = cancelCause !== 'none'` _before_ calling `abortController.abort()`. The wire-level `abort()` still happens for downstream cleanup, but it no longer pollutes telemetry.
   - Pros: keeps the existing one-place flush, distinguishes user-cancel from shutdown, leaves the `AbortController` as the single source for downstream cancellation.
   - Cons: introduces one extra piece of state; needs care so every "I'm exiting because of abort" branch sets the cause.

2. **Flush telemetry _before_ `abortController.abort()`.**
   Move `flushCompletionEvent()` above the `abort()` call in the `finally`.
   - Pros: minimal diff, two lines reordered.
   - Cons: only works because both calls are synchronous; brittle if anyone later adds awaits or rearranges cleanup. Does not distinguish "framework stopped me" from "I finished cleanly" ŌĆö both will appear as `aborted='false'` because the controller has not yet been aborted by the cleanup itself.

3. **Replace the boolean `aborted` with a three-valued `outcome` ('success'|'cancel'|'error').**
   Set `outcome` explicitly at each terminal point (after the final yield, in the abort listener, in a `catch`).
   - Pros: better semantics; clears up F7 (error-path telemetry gap) at the same time.
   - Cons: dashboards that already filter on `aborted='true'/'false'` need updating; larger blast radius.

> Combine #1 and #3 for the cleanest fix.

**Resolution:** Chose option **#3** (three-valued `outcome`), with the `outcome` stamped BEFORE cleanup runs ŌĆö which also subsumes option #1. Default the variable to `'cancel'` so a generator that exits early without reaching the final `yield` is recorded as a cancel; promote to `'success'` after the terminal `yield` lands; (F7 then sets `'error'` in a `catch`). For dashboard back-compat the original `aborted` property is retained as a derived value (`outcome === 'cancel'`). Commit: [`639935df`](https://github.com/microsoft/vscode-documentdb/pull/711/commits/639935df). Rationale: the boolean was semantically wrong (it conflated cancel-by-user, cancel-by-shutdown, and ŌĆ£cleanup aborted usŌĆØ), and once we were touching this anyway, F7's error path needed a third value as well ŌĆö doing both at once kept the change contained to one place.

### F2 ŌĆö Background Query Insights prefetch is skipped after a query reset

**Original severity:** Medium
**My severity:** **Medium (confirmed)** ŌĆö not data loss (the fallback fetch inside the Query Insights tab still recovers), but it silently defeats the perceived-responsiveness goal of the PR for the common "run query, stay on Results, open Insights later" flow. Severity stays Medium because there is a working fallback.

**Where:** [src/webviews/documentdb/collectionView/CollectionView.tsx](../../../../src/webviews/documentdb/collectionView/CollectionView.tsx#L143-L158), [src/webviews/documentdb/collectionView/CollectionView.tsx](../../../../src/webviews/documentdb/collectionView/CollectionView.tsx#L161-L209), [src/webviews/documentdb/collectionView/CollectionView.tsx](../../../../src/webviews/documentdb/collectionView/CollectionView.tsx#L301-L344)

**Verification:** Confirmed. Both effects key on `[currentContext.activeQuery]`, so they fire in the same render's effect phase, both closed over the render's `currentContext`. The reset effect schedules a state update that resets `queryInsights` to `idle`. The query-running effect dispatches `runFindQuery` and then calls `prefetchQueryInsights()` from its `.then()`. `prefetchQueryInsights()` reads `currentContext.queryInsights` _from that closure_ ŌĆö the pre-reset value. If the previous run reached `s1Success` / `s2Success` / `s3Success` / `s3Idle`, the early `if (qi.kind !== 'idle') return;` aborts the prefetch. The reducer's later auto-reset is invisible to the stale closure.

**Proposed solutions:**

1. **Move the claim into a functional `setCurrentContext` updater** (recommended).
   Inside `prefetchQueryInsights`, call `setCurrentContext((prev) => ...)`; inside the updater, look at `prev.queryInsights`; if it is not `idle`, return `prev` unchanged and abort the request via a captured boolean. Only fire `getQueryInsightsStage1.query()` when the claim succeeded.
   - Pros: race-free, reads the latest state without a ref, idiomatic React. Same place still owns the dedupe contract documented in the comment.
   - Cons: slightly awkward to thread "did we claim it?" out of a functional updater; usual pattern is to do the claim _and_ schedule the request after `setState` returns, using a microtask or `useEffect` on the new `queryInsights` state.

2. **Read latest state via a ref.**
   The component already maintains `currentContextRef`. Read `currentContextRef.current.queryInsights` instead of `currentContext.queryInsights` inside `prefetchQueryInsights`.
   - Pros: one-line change.
   - Cons: refs lag effects by one render, so this only fixes the closure issue when the reset effect has already committed by the time the network call resolves ŌĆö which is usually but not always true (depends on timing of `runFindQuery`). Still a meaningful improvement, but not as airtight as solution #1.

3. **Merge the reset and prefetch into a single effect.**
   When `executionIntent` is `'initial'` or `'refresh'`, reset Query Insights state, then dispatch `runFindQuery`, then `prefetchQueryInsights` ŌĆö all in one effect, all using a freshly-built `queryInsights = DefaultCollectionViewContext.queryInsights` local variable.
   - Pros: removes the dependency on intermediate state altogether; clearest intent.
   - Cons: harder refactor; risks regressing the existing separation that the comment relies on.

**Resolution:** Initially attempted option **#1** in [`4d398dd1`](https://github.com/microsoft/vscode-documentdb/pull/711/commits/4d398dd1) ŌĆö **reverted** because it broke the prefetch entirely. The fix landed as option **#2** (ref-based read) in [`4a31ac18`](https://github.com/microsoft/vscode-documentdb/pull/711/commits/4a31ac18).

Why #1 failed in practice: `setState((prev) => ...)` **queues** the updater ŌĆö the updater runs on React's next render, not synchronously inside the call. The captured `claimed` boolean was therefore always `false` when the `if (!claimed) return;` line executed, so the network call never fired. Worse, the updater itself still committed, leaving state stuck on `s1Loading`; the Query Insights tab's fallback fetch then saw "someone else is loading" and waited forever. This is the standard "side-effect from setState callback" anti-pattern; the StrictMode-purity caveat I noted was the right shape of concern but the wrong root cause.

Why #2 works here in practice despite its caveat: by the time `runFindQuery.then` resolves (a network call ŌĆö tens to hundreds of ms), React has long since committed the reset effect's state update *and* the `currentContextRef`-updating effect has copied it into the ref. The one-commit lag I called out earlier is not observable for this caller because many commits happen between the reset and the `.then`. If a future caller invokes `prefetchQueryInsights` synchronously from inside the render that queues the reset, the lag could bite ŌĆö at that point the right escalation is option #1 done properly (a request-key on Stage 1, analogous to Stage 3) rather than another setState-callback trick.

**Original severity:** Medium
**My severity:** **Medium (confirmed, but narrower than originally stated)** ŌĆö only the _placeholder_ slots inside `MarkdownCard` and the `ImprovementCardShell` actually render `StreamingInlineProgress` (it is the "still workingŌĆ”" body when no markdown / no rec content has arrived yet). At Stage 3 click, this is in fact 3 placeholders + 1 Announcer mounted in the same paint. The screen-reader spam claim is real, but it's worst during the connecting/submitted window before the first structured event arrives; once content streams in, most placeholders are replaced and the noise subsides.

**Where:** [src/webviews/documentdb/collectionView/components/queryInsightsTab/QueryInsightsTab.tsx](../../../../src/webviews/documentdb/collectionView/components/queryInsightsTab/QueryInsightsTab.tsx#L640-L781), [src/webviews/documentdb/collectionView/components/queryInsightsTab/components/streamingPlaceholder/StreamingInlineProgress.tsx](../../../../src/webviews/documentdb/collectionView/components/queryInsightsTab/components/streamingPlaceholder/StreamingInlineProgress.tsx#L59-L73), [src/webviews/documentdb/collectionView/components/queryInsightsTab/components/optimizationCards/custom/Stage3AnalyzingCard.tsx](../../../../src/webviews/documentdb/collectionView/components/queryInsightsTab/components/optimizationCards/custom/Stage3AnalyzingCard.tsx#L155-L168)

**Verification:** Confirmed. `StreamingInlineProgress` hard-codes `role="status" aria-live="polite"`. `QueryInsightsTab` pre-reserves three slots that may each render it (analysis, first recommendation shell, educational), and `Stage3AnalyzingCard` mounts an `Announcer` saying "AI is analyzingŌĆ”". On Stage 3 start, all four regions enter the DOM in the same paint. Polite live regions queue rather than interrupt, so this is more "verbose noise" than "users miss content" ŌĆö still a regression versus the previous review's removal of the dead `StreamingPlaceholder`.

**Proposed solutions:**

1. **Default `StreamingInlineProgress` to non-live; add opt-in `announce` prop** (recommended).
   Render `<div className=...>` (no `role`/`aria-live`). Add `announce?: boolean` and use it only at the single point where a global "AI is working" signal is wanted ŌĆö which is already covered by `Stage3AnalyzingCard`'s `Announcer`.
   - Pros: keeps the visible spinner exactly as today; eliminates duplicate announcements; small surface area.
   - Cons: requires touching every caller of `StreamingInlineProgress` to confirm none of them need the announcement (today none do ŌĆö `Stage3AnalyzingCard` already covers it).

2. **Use `role="progressbar"` (indeterminate) instead of `role="status"`.**
   `role="progressbar"` with `aria-valuetext` describes ongoing work without being a live region.
   - Pros: semantically more accurate ("activity indicator" not "status message"); screen readers handle progress widgets quietly.
   - Cons: progressbar without min/max is unusual; some screen readers still announce on mount; less direct fix than #1.

3. **Single global `aria-live` region in the tab, fed by stream phase only.**
   Replace per-slot live regions and the `Announcer` with one tab-level live region that announces "AnalyzingŌĆ”", "Receiving recommendationsŌĆ”", "Done" at phase transitions.
   - Pros: one announcement source, predictable cadence; matches the streaming reducer's phase model.
   - Cons: largest change; needs careful UX wording for assistive tech.

**Resolution:** Chose option **#1** (non-live by default with opt-in `announce` prop). Commit: [`399766b0`](https://github.com/microsoft/vscode-documentdb/pull/711/commits/399766b0). Rationale: smallest surface area that actually solves the spam, keeps the visible spinner unchanged, and no current call site needs the announcement (`Stage3AnalyzingCard`'s `Announcer` is the single global signal). Pairs with F9 to give screen-reader users a phase-aware narrative from one source instead of overlapping per-slot ones. Option #2 (`role="progressbar"`) was rejected because progressbar without min/max is unusual and some screen readers still announce on mount; option #3 (single tab-level live region) is the right long-term shape but a larger change than this PR's scope warrants.

### F7 ŌĆö `streamStage3` has no `catch`; error-path telemetry is lost and reports as cancel

**Severity:** Medium

**Where:** [src/webviews/documentdb/collectionView/queryInsights/queryInsightsEventsRouter.ts](../../../../src/webviews/documentdb/collectionView/queryInsights/queryInsightsEventsRouter.ts#L186-L572)

**Observation:** `streamStage3`'s async generator body has only `try { ... } finally { ... }`. If `streamHandle.fragments` throws mid-iteration, or `streamHandle.completion` rejects, the error propagates to the framework and surfaces as `onError` on the webview side. The `finally` still fires, but:

- It writes `aborted='true'` (because of F1 _and_ because the controller is aborted in cleanup), so error outcomes are indistinguishable from cancels.
- No error metadata is captured (`errorKind`, `errorMessage`, partial counts other than the ones already recorded before the throw).

The dedicated `STAGE3_COMPLETION_EVENT` is documented as the canonical Stage 3 telemetry; right now error analytics for streaming Stage 3 is effectively blind.

**Proposed solutions:**

1. **Add a `catch (err)` that records `outcome='error'` + `errorKind` / `errorMessage`, then rethrows.** Pair this with F1 solution #3 (three-valued outcome) so success/cancel/error are first-class.
   - Pros: closes the analytics gap created by F1; tiny diff.
   - Cons: must rethrow so the framework's `onError` still fires for the webview reducer.

2. **Leave error-path telemetry to the auto rpc event.**
   - Pros: zero new code.
   - Cons: the auto rpc event is exactly what this dedicated event was created to _replace_ (plan ┬¦7 / WI-10), so this is a regression of the PR's stated goal.

**Resolution:** Chose option **#1** (`catch` block records `outcome='error'` + metadata, then rethrows). Commit: [`8a44403c`](https://github.com/microsoft/vscode-documentdb/pull/711/commits/8a44403c). Rationale: the F1 refactor already introduced the three-valued `outcome` and the `errorKind`/`errorMessage` slots cost effectively nothing once that's in place. The rethrow preserves the framework's `onError` path so the webview reducer still flips to its failure state. Option #2 (defer to the auto rpc event) would have undone the PR's stated WI-10 goal of making the dedicated event the canonical Stage 3 telemetry.

### F8 ŌĆö `prefetchQueryInsights` failure callback can clobber a newer request's state

**Severity:** Low

**Where:** [src/webviews/documentdb/collectionView/CollectionView.tsx](../../../../src/webviews/documentdb/collectionView/CollectionView.tsx#L191-L209)

**Observation:** The `.catch()` handler unconditionally dispatches `stage1Failed(prev.queryInsights, ...)`. If the user has already started a _new_ query (so `prev.queryInsights` is in `s1Loading` for a different request, or has progressed past it), the old failure will be folded into the current state machine, producing a spurious error display or a transition the reducer did not expect. The reducer's request-key guards do not extend to Stage 1 ŌĆö only Stage 3 uses `requestKey` for late-event filtering.

**Proposed solutions:**

1. **Carry a per-call generation token (similar to `stage3RequestKey`) into Stage 1.**
   In the `.then` / `.catch`, only commit if the token still matches `prev.queryInsights.requestKey`.
   - Pros: matches the proven Stage 3 pattern; closes the race fully.
   - Cons: requires reducer plumbing for a previously-untracked field.

2. **Bail out if `prev.queryInsights.kind !== 's1Loading'` inside the `.catch`.**
   - Pros: minimal change.
   - Cons: still races with a _second_ `s1Loading` (different request).

**Resolution:** Chose option **#2** (kind-based bail-out in both `.then` and `.catch`). Commit: [`4c710571`](https://github.com/microsoft/vscode-documentdb/pull/711/commits/4c710571). Rationale: Stage 1 is short-lived enough in practice (sub-second explain `queryPlanner`) that the residual second-`s1Loading` race is acceptable for now. Option #1 (request-key plumbing through the reducer) is the right shape if/when Stage 1 gains longer-running variants, but the engineering cost is disproportionate to the actual exposure today.

### F10 ŌĆö `CardStack` in-render `setState` is correct but load-bearing on an off-component `useMemo`

**Severity:** Low (informational; tied to Copilot thread #3)

**Where:** [src/webviews/documentdb/collectionView/components/queryInsightsTab/components/cardStack/CardStack.tsx](../../../../src/webviews/documentdb/collectionView/components/queryInsightsTab/components/cardStack/CardStack.tsx#L85-L92)

**Observation:** The previous response to Copilot correctly noted there is no infinite loop ŌĆö the `items !== lastNonEmpty` guard converges in the very next render. With `insightCards` memoized in `QueryInsightsTab`, the cost is bounded to "one extra render per data change", which is acceptable. The fragility is that any future contributor who removes the `useMemo` wrapping `insightCards` will reintroduce per-render churn without it being obvious from `CardStack` alone.

**Proposed solutions:**

1. **Replace the in-render setter with a `useEffect` that updates `lastNonEmpty` after commit.**
   - Pros: eliminates the extra render and the StrictMode double-invocation concern; matches the "store-derived state with effect" idiom most React codebases use.
   - Cons: the snapshot lags by one render ŌĆö during the exit animation's first frame, `renderedItems` would briefly be `items` (now empty) before the effect runs. The current pattern was chosen deliberately to avoid this; only switch if you confirm the exit animation still looks right.

2. **Use `useSyncExternalStore`-style "derived from props" with `useState` + `useMemo`-tracked sentinel.**
   - Pros: also avoids in-render setState.
   - Cons: more code than today, same observable behavior.

3. **Document the load-bearing `useMemo` requirement in a single-line comment in `QueryInsightsTab` _and_ `CardStack`** (recommended).
   - Pros: zero behavior change; addresses the only practical risk (future refactor breaking it).
   - Cons: relies on the comment surviving.

**Resolution:** Chose option **#3** (cross-referencing ŌĆ£LOAD-BEARINGŌĆØ comments on both sides). Commit: [`b22c9478`](https://github.com/microsoft/vscode-documentdb/pull/711/commits/b22c9478). Rationale: the pattern is correct (no infinite loop ŌĆö the convergence argument from the earlier Copilot thread stands), so the cost-benefit of options #1/#2 (refactor to `useEffect` or `useSyncExternalStore`-style) doesn't pay off. The only real risk is a future refactor dropping the off-component `useMemo` without realizing `CardStack` depends on it; explicit comments on both sides eliminate that risk at zero behavior cost.
