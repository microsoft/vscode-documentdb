# Step 6.1 — Shared Schema Cache (SchemaStore)

**PR:** [#538](https://github.com/microsoft/vscode-documentdb/pull/538)

## Summary

Introduced a `SchemaStore` singleton that accumulates schema data per `{clusterId, databaseName, collectionName}` triple, enabling schema sharing across Collection View tabs and between the Collection View and the Query Playground.

## Problem

Before `SchemaStore`, schema was siloed:

- Two Collection View tabs open on the same collection had independent `SchemaAnalyzer` instances — schema work was duplicated
- The Query Playground had no access to schema data at all — no field completions
- Schema was owned by `ClusterSession`, which was tied to a webview tab — closing a tab lost all schema knowledge
- Query Playground execution results went unanalyzed — documents returned by `db.users.find()` in the playground contributed nothing to schema

## Solution

A shared, cluster-scoped schema cache at `src/documentdb/SchemaStore.ts`:

```
SchemaStore (singleton, keyed by clusterId::db::collection)
  ├── users@cluster1/myDb      → SchemaAnalyzer (accumulated from ALL sources)
  ├── orders@cluster1/myDb     → SchemaAnalyzer
  └── products@cluster1/myDb   → SchemaAnalyzer
```

### Key Design Decisions

| Decision       | Choice                                      | Rationale                                                            |
| -------------- | ------------------------------------------- | -------------------------------------------------------------------- |
| Location       | `src/documentdb/SchemaStore.ts`             | Follows existing cache pattern (`ClustersClient`, `CredentialCache`) |
| Cache key      | `{clusterId}::{db}::{collection}`           | Stable `clusterId`, `::` separator avoids collision                  |
| Ownership      | SchemaStore owns `SchemaAnalyzer` instances | Enables centralized events and metadata tracking                     |
| Events         | Debounced `onDidChangeSchema` (1s per key)  | Prevents excessive churn during rapid page navigation                |
| ClusterSession | Delegates entirely to SchemaStore           | Single source of truth, no duplication                               |

### Schema Feeding

- **Collection View:** `ClusterSession.addDocuments()` writes to SchemaStore (reads via `getKnownFields()`)
- **Query Playground:** Execution results with a known namespace (`result.source.namespace`) are fed to SchemaStore — running `db.users.find()` in the playground populates field completions for `users` across all surfaces
- **Lifecycle cleanup:** Connection removal clears cluster data; collection/database deletion clears corresponding schemas

## Impact

| Scenario                    | Before              | After                                         |
| --------------------------- | ------------------- | --------------------------------------------- |
| Two tabs on same collection | Independent schemas | Shared — Tab B gets Tab A's field completions |
| Playground `find()` result  | Documents discarded | Fed to SchemaStore → completions available    |
| Tab closed, new tab opened  | Schema lost         | Schema persists in SchemaStore                |
| Connection removed          | Schema leaked       | Schema cleared via `clearCluster()`           |

## Testing

22 unit tests covering read/write operations, key isolation, event debouncing, lifecycle management, and singleton behavior.
