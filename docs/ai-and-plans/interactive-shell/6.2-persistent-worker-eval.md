# Step 6.2 — Persistent Worker Eval

**PR:** [#540](https://github.com/microsoft/vscode-documentdb/pull/540)

## Summary

Replaced the in-process `vm.runInContext()` evaluator with a persistent worker thread (Option F) that owns its own database client. This provides infinite-loop safety, client isolation, and zero re-authentication overhead after the first run.

## Motivation

The in-process evaluator (Step 6, Option A) had two fundamental problems:

### Problem 1: Synchronous Infinite Loops Freeze the Extension

`Promise.race()` + `setTimeout()` only fires if the event loop is free. A synchronous `while(true){}` blocks the extension host permanently. The `vm.timeout` option doesn't help because `@mongosh`'s async rewriter transforms user code into async IIFEs, and `vm.timeout` doesn't apply to async execution.

### Problem 2: Shared MongoClient State

The Query Playground and Collection View shared the same `MongoClient`. Long-running playground queries could exhaust pool connections, and a disconnect/reconnect cycle from the playground would affect Collection View browsing.

## Options Evaluated

| Option              | Auth Overhead         | Isolation   | Infinite Loop Safety  | Resource Waste      |
| ------------------- | --------------------- | ----------- | --------------------- | ------------------- |
| A (In-process)      | None                  | ❌ Shared   | ❌ Freezes extension  | None                |
| B (Per-run worker)  | Per execution         | ✅ Full     | ✅ Kill worker        | Per-run cost        |
| E (Eager worker)    | Once on connect       | ✅ Full     | ✅ Kill & respawn     | Always-on worker    |
| **F (Lazy worker)** | **Once on first Run** | **✅ Full** | **✅ Kill & respawn** | **None until used** |

**Option F selected** — same isolation and safety benefits as Option E, but avoids creating a worker for users who never use the Query Playground. Aligns with the extension's lazy-loading philosophy.

## Architecture

```
Extension Host (Main Thread)              Worker Thread (lazy, persistent)
├── ScratchpadEvaluator                   ├── Own MongoClient (auth once)
│   ├── evaluate(connection, code)        ├── ShellInstanceState
│   ├── ensureWorker() → lazy spawn      ├── ShellEvaluator
│   ├── shutdown() → graceful close       └── vm.runInContext(code)
│   └── killWorker() → force terminate
```

### IPC Protocol

Custom typed message protocol using discriminated unions:

- **Main → Worker:** `init` (connection + auth), `eval` (code + options), `shutdown`, `tokenResponse`/`tokenError`
- **Worker → Main:** `initResult`, `evalResult`/`evalError`, `shutdownComplete`, `tokenRequest`, `log`, `consoleOutput`

### Discussion: Entra ID Authentication in the Worker

The worker cannot use VS Code's `authentication.getSession()` directly (no VS Code API in workers). Solution: the worker sends a `tokenRequest` IPC message, the main thread acquires the token via VS Code's auth API, and replies with `tokenResponse`. This reuses the existing VS Code session without re-prompting the user.

## Key Implementation Details

- **Canonical EJSON** (`relaxed: false`) for IPC — preserves Int32/Long/Double type fidelity for schema analysis
- **Init failure recovery** — `spawnWorker()` calls `terminateWorker()` on failure, returning to a respawnable state
- **No feature flag** — direct migration; old in-process eval path removed entirely
- **Telemetry** — `scratchpad.execute` event with session tracking, auth method, run mode, result type, timing

## Bundle Impact

- `main.js`: ~6.7 MB (extension host)
- `scratchpadWorker.js`: ~6.8 MB (worker thread, separate webpack entry point)

## Testing

All 30 existing tests pass. Manual testing covered SCRAM auth, infinite loop recovery, cancellation, editor close shutdown, and schema stats.
