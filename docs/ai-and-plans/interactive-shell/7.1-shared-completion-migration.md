# Step 7.1 — Shared Completion Migration

**PR:** [#551](https://github.com/microsoft/vscode-documentdb/pull/551)

## Summary

Pure structural refactoring — no functionality changes. Established a clean organizational boundary for query-language completion code by migrating shared logic out of the `webviews/` tree.

## Motivation

After Step 7, platform-neutral completion logic (cursor context detection, snippet utilities, sort prefix definitions, type suggestion data) lived under `src/webviews/documentdbQuery/shared/`. This was semantically incorrect — these modules are imported by extension-host code (the playground provider) and have no dependency on Monaco or webview APIs.

## Three-Phase Migration

| Phase | Move                                                                                       | Files                          |
| ----- | ------------------------------------------------------------------------------------------ | ------------------------------ |
| 1–5   | `webviews/documentdbQuery/shared/` → `documentdb/query-language/shared/`                   | 11 files (platform-neutral)    |
| 6     | `documentdb/scratchpad/completions/` → `documentdb/query-language/scratchpad-completions/` | 8 files (VS Code API specific) |
| 7     | `webviews/documentdbQuery/` → `webviews/query-language-support/`                           | 25 files (folder rename)       |

## Final Directory Structure

```
src/documentdb/query-language/
  shared/                         # Platform-neutral core (no Monaco, no VS Code)
  scratchpad-completions/         # VS Code API specific (extension host)

src/webviews/query-language-support/  # Monaco specific (webview)
```

All moves used `git mv` to preserve history (98–100% similarity detected). Import paths updated across 20 files. No path aliases needed (deepest relative path is 3 levels).
