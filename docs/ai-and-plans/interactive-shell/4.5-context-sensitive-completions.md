# Step 4.5 — Context-Sensitive Completions

**PR:** [#530](https://github.com/microsoft/vscode-documentdb/pull/530)

## Summary

Added cursor-position-aware completion filtering to the `documentdb-query` language. Instead of showing a flat list of all operators at every cursor position, completions are now filtered and ranked based on where the cursor sits in the query expression.

## Motivation

Before this step, the completion provider returned the same list everywhere — field names, query operators, BSON constructors all appeared regardless of cursor position. Existing MongoDB API tooling returns the same flat list. We made the deliberate choice to go further and provide context-sensitive completions as a differentiating feature.

## Cursor Context Detection

A heuristic backward-scanning algorithm determines the cursor's semantic position:

| Context               | Example                 | What Shows                                                    |
| --------------------- | ----------------------- | ------------------------------------------------------------- |
| **Key** (root)        | `{ \| }`                | Field names + logical operators (`$and`, `$or`, `$nor`)       |
| **Value**             | `{ age: \| }`           | Type suggestions + operators + BSON constructors + JS globals |
| **Operator** (nested) | `{ age: { \| } }`       | Comparison operators with brace-stripped snippets             |
| **Array element**     | `{ $and: [ \| ] }`      | Same as key position                                          |
| **Unknown**           | Ambiguous/garbled input | Full list (safe fallback)                                     |

### Discussion: Why Heuristic Over AST?

We evaluated a "dummy completion" strategy — inserting a placeholder at the cursor to make the expression parseable by `acorn`, then inspecting the AST. This was rejected because:

1. **Chicken-and-egg problem** — the placeholder shape depends on context, which is what we're trying to determine
2. **Multiple parse errors** — real mid-typing states often have several syntax issues; patching one doesn't fix the rest
3. **Performance** — string surgery + full parse on every keystroke vs. O(n) backward scan
4. **Maintenance** — repair heuristics would be tightly coupled to `acorn`'s error behavior

The backward scanner handles all common partial-input states reliably and falls back to "show everything" for genuinely ambiguous positions.

## Type-Aware Value Suggestions

When the field's BSON type is known from the schema, contextual value suggestions appear first:

| Field Type | Suggestions                                     |
| ---------- | ----------------------------------------------- |
| Boolean    | `true`, `false`                                 |
| Number     | Range query `{ $gt: …, $lt: … }`, minimum value |
| String     | Regex pattern, string literal                   |
| Date       | `ISODate("…")`, date range, last N days         |
| ObjectId   | `ObjectId("…")`                                 |
| Array      | `{ $elemMatch: { … } }`, `{ $size: … }`         |
| Null       | `null`                                          |

## Type-Aware Operator Ordering

All operators remain visible — nothing is hidden. Sort prefixes control ranking:

- `0_` — Type-relevant (operator's `applicableBsonTypes` matches the field)
- `1a_` — Comparison operators (universal, most common)
- `1b_` — Other universal operators
- `2_` — Non-matching (demoted, not hidden)
- `3_` — BSON constructors
- `4_` — JS globals

### Discussion: Why Not Hide Non-Matching Operators?

Fields may be polymorphic — `age` might be `Number` 98% of the time but occasionally `String`. Hiding `$regex` for a Number field would be incorrect. Demoting it to the bottom preserves discoverability while keeping relevant operators front and center.

## JavaScript Global Completions

Restored JS globals that were lost when the custom `documentdb-query` language replaced the Monaco JS worker: `Date`, `Date.now()`, `Math.floor()`, `RegExp`, `Infinity`, `NaN`, `undefined`. These are sourced from `@mongodb-js/shell-bson-parser`'s sandbox scope — the same runtime that executes queries.

## Code Organization

The monolithic completion provider was refactored into a `completions/` folder with focused modules: `createCompletionItems.ts`, `mapCompletionItems.ts`, `typeSuggestions.ts`, `jsGlobals.ts`, `completionKnowledge.ts`, `snippetUtils.ts`.

## Testing

202 tests across 5 suites — 41 cursor context tests covering both complete and incomplete input, 95 completion provider tests, 38 validator tests.
